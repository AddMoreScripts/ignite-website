Ignite vs Redis

Both, Apache Ignite and Redis provide distributed caching functionality, but the features supported by either product are very different. Redis is mainly a data structure store, while Ignite fabric provides many in-memory distributed components including data grid, compute grid, streaming, and also data structures.

Given that Ignite is an in-memory data fabric and offers more functionality, a complete feature-by-feature comparison is not really relevant, while we can still compare some individual data grid features.

Apache Ignite Redis
JCache (JSR 107)

Ignite is fully compliant with JCache (JSR 107) specification for caching.

Redis does not implement JCache (JSR 107) specification for caching.

ACID Transactions

Ignite has full support for ACID transactions, including OPTIMISTIC and PESSIMISTIC concurrency modes, as well as READ_COMMITTED, REPEATABLE_READ, and SERIALIZABLE isolation levels.

Redis provides very limited support for client-side OPTIMISTIC transactions. In case of concurrent update, clients are required to manually retry a transaction.

Data Partitioning

Ignite supports PARTITIONED caches which is similar to a distributed hash map with every node in the cluster responsible only for a portion of the data. Ignite will automatically rebalance the data in case of topology changes.

Redis does not provide partitioning, but it provides sharding with replicas. Working with shards is more rigid and almost always requires a set of fairly involved manual steps on both, clients and servers, every time a topology changes.

Full Replication

Ignite has support for REPLICATED caches where every key-value pair is replicated on every node in the cluster.

Redis does not have direct support for full replication.

Native Objects

Ignite allows users to use their own domain object model and provides native support for any Java/Scala, C++, and .NET/C# data types (objects). Users are free to store any application and domain object in Ignite caches.

Redis does not allow users to work with their own data types and only supports the predefined set of primitive data structures, such as sets, lists, arrays, and some others.

Client-side (Near) Caching

Ignite provides direct support for client-side caching for most recently accessed data.

Redis does not provide support for client-side caching.

Collocated (server-side) Processing

Ignite allows executing any native Java, C++, and .NET/C# code directly on the server-side, close to the data, in collocated fashion.

Redis generally does not have any collocated processing and allows primitive server-side code execution only with LUA scripting language. There is no direct support for Java, .NET, or C++ on the server side.

SQL Queries

Ignite supports complete SQL (ANSI-99) syntax for querying in-memory data.

Redis does not have any query language and only supports client-side caching API.

Continuous Queries

Ignite provides direct support for client-side and server-side continuous queries. Users can setup server-side filters to reduce and narrow down the amount of events sent to clients.

Redis provides support for client-side key-based event notifications. However, it does not allow to provide server-side filters, hence causing a significant network traffic for update notifications between servers and clients.

Database Integration

Ignite can automatically integrate with external databases - RDBMS, NoSQL, and HDFS.

Redis does not integrate with external databases.
