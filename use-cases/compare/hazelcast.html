Ignite vs Hazelcast Data Grid

Both, Apache Ignite and Hazelcast provide a feature rich data grid functionality responsible for partitioning and caching data in memory with the capability to scale out across distributed clusters.

Feature Comparison

Apache Ignite and Hazelcast have many differences in the way caching, transactions, and data querying are supported. Below we outline only major differences, which in our opinion are most important whenever choosing an in-memory data grid product.



Apache Ignite Hazelcast
Open Source Focus

Ignite has been growing and enhancing its open source feature set. The latest major additions include Ignite Fabric for C++ and .NET/C#, with Node.JS integration coming soon as well.

Hazelcast has continuously been removing features from its open source offering and moving them to enterprise (paid) edition. This is true for features like off-heap memory, continuous queries, web-session clustering, SSL encryption support, and more.

JCache (JSR 107)

Ignite is fully compliant with JCache (JSR 107) specification for caching.

Hazelcast is fully compliant with JCache (JSR 107) specification for caching.

Off-Heap Memory

Ignite supports storing data in on-heap or off-heap memory, depending on user configuration.

Hazelcast supports off-heap memory only in its commercial (paid) offering.

Off-Heap Indexes

Whenever configured with off-heap memory, Ignite will store query indexes off-heap as well (in order not to affect on-heap memory used by user applications).

Hazelcast does not have support for off-heap indexes.

Continuous Queries

Ignite provides support for continuous queries, i.e. allows clients and servers to subscribe for continuous notifications to the data updates.

Hazelcast offers support for continuous queries only in its commercial (paid) edition.

SSL Encryption

Ignite provides support for SSL/TLS encryption, i.e. allows all clients and servers to communicate over encrypted SLL or TLS protocols.

Hazelcast offers support for SLL encryption only in its commercial (paid) edition.

SQL Queries

Ignite supports complete SQL (ANSI-99) syntax for querying in-memory data.

Hazelcast has very limited support for SQL (only a few keywords).

Queries with JOINs

Ignite SQL includes full support for SQL JOINs, including JOINs across multiple caches, for example:

   select * from A a, B b where a.b_id = b.id

Hazelcast does not have any support for JOIN queries (with or without SQL). Whenever needed, users perform JOINs by manually combining multiple query results.

Query Consistency

Ignite provides full query consistency, i.e. queries are executed on a certain snapshot of data, and updates that happened after the query execution started do not affect the query result.

Hazelcast queries are inconsistent and it is possible that part of the query result will see a certain update and another part will not.

Query Fault Tolerance

Ignite queries are fault-tolerant, i.e. query result is always consistent and is not affected by cluster topology changes, like nodes joining, leaving, or crashing.

Hazelcast queries are not fault-tolerant, i.e. query results become inconsistent during cluster topology changes, while the data is being re-balanced in the background.

Data Consistency

Ignite supports atomic and transactional consistency on all in-memory data, regardless of whether data is stored in PARTITIONED or REPLICATED caches.

Hazelcast supports atomic and transactional consistency only on PARTITIONED caches. Data stored in REPLICATED caches does not have any transactional guarantees at all.

Deadlock-Free Transactions

Ignite supports deadlock-free, optimistic transactions, which do not acquire any locks, and free users from worrying about the lock order. Such transactions also provide much better performance.

With Hazelcast, you always need to worry about updating data in the same order to avoid deadlocks, which is often impossible, especially in large projects.

Transactional EntryProcessor

Ignite supports transactional EntryProcessor, which allows to execute collocated user logic on the server side, within a transaction.

With Hazelcast, it is impossible to use EntryProcessor or execute user logic within a transaction.

Web Session Clustering

Ignite supports caching and clustering web-sessions for all known application servers.

Hazelcast offers web-session clustering only in its commercial (paid) edition.

Compute Grid

Ignite provides support for map-reduce, fork-join, and basic distributed lambda processing on the cluster, including job load-balancing, fault-tolerance, checkpointing, scheduling, etc.

Hazelcast only provides support for map-reduce and random job distribution within a cluster.

Streaming Grid

Ignite provides support for in-memory streaming, including support for maintaining and querying sliding windows of streaming data.

Hazelcast does not offer any support for streaming.

Service Grid

Ignite allows users to easily cluster-enable their services, including support for various cluster-singletons.

Hazelcast managed services do not provide support for cluster-singletons.

Fabric for .NET/C#

Ignite provides full in-memory fabric APIs for .NET/C# users, including ability to execute C# closures, C# caching, transaction, and querying APIs, native C# CacheStore APIs, etc.

Hazelcast provides .NET/C# limited client APIs only.

Fabric for C++

Ignite provides full in-memory fabric APIs for C++ users.

Hazelcast provides C++ limited client APIs only.

Fabric for Node.JS

Ignite provides client API support for Node.JS users (coming in 1.5).

Hazelcast does not have any support for Node.JS.
